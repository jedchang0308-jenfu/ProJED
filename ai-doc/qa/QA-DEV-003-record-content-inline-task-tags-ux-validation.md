# QA-DEV-003：紀錄內容 inline 任務 tag UX 驗證計畫

狀態：Done / Static QC Covered
日期：2026-06-04  
Owner：QA / PM  
關聯 DEV：DEV-003  
關聯規格：`ai-doc/specs/SPEC-004-record-content-inline-task-tags.md`

## 1. 驗證目的

本驗證計畫用使用者視角確認 DEV-003 的 UX 是否真的順暢：

- 使用者在寫會議紀錄或個人工作紀錄時，可以像在 Codex 對話框 tag skill 一樣，把任務插入內容。
- 使用者不需要離開熟悉的看板，也不會被帶到另一個陌生選取頁。
- 任務 tag 插入位置符合內容游標位置，而不是固定加在文末或只出現在關聯任務區。
- 右側紀錄欄在選取任務時自動收起，讓使用者能在原本看板上直接點選任務。
- 內容中的任務 tag 與結構化 `record_task_links` 能維持一致，讓後續 AI 全域分析有穩定資料來源。

## 2. 驗證範圍

### In Scope

- 會議紀錄填寫流程。
- 個人工作紀錄填寫流程。
- 右側紀錄欄展開、收合、回復。
- 從看板直接選取任務。
- Kanban card 任務與 checklist 子任務選取。
- 內容輸入框中的 inline task tag 顯示、插入、重複引用、刪除。
- 關聯任務摘要清單與內容 tag 的一致性。
- 草稿儲存、發布、重新開啟後的呈現。
- Desktop、laptop、小視窗 viewport 下的遮擋、溢出、焦點遺失。

### Out of Scope

- 新增資料庫 schema。
- rich text editor 全功能，例如粗體、列表、多人協作編輯。
- member/user mention chip。
- RAG embedding pipeline 的深度語意品質驗證。

## 3. 測試前提與資料

測試環境：

- URL：`http://127.0.0.1:4174/`
- 測試看板需至少有一個可進入的 board。
- 若目前 Supabase migration 尚未套用，`knowledge_records` 或 `board_role_permissions` 404 可列為環境阻塞，但仍可先驗證 draft UI flow。

測試資料：

- 一張短標題任務卡，例如 `資料庫改 SQL`。
- 一張長標題任務卡，驗證 tag 與卡片內容是否截斷或換行合理。
- 一個 checklist 子任務。
- 一張不同狀態欄位中的任務卡。
- 會議紀錄內容文字：`今天討論 release 風險與後續追蹤。`
- 個人工作紀錄內容文字：`上午處理測試資料，下午整理紀錄關聯。`

建議 viewport：

- Desktop：`1440x900`
- Laptop：`1366x768`
- Small：`390x844` 或瀏覽器窄窗

## 4. 使用者關鍵流程

1. 使用者在看板中工作，想到需要補一筆會議紀錄。
2. 使用者開啟右側紀錄欄，填入參與人員與內容。
3. 使用者寫到一半，需要提到某個任務，點擊 `從看板選取`。
4. 系統收起右側欄，保留原本 Kanban 看板，進入任務選取模式。
5. 使用者直接點看板上的任務卡或 checklist 子任務。
6. 使用者完成選取後回到紀錄欄。
7. 系統把任務以 tag/chip 形式插入內容游標位置。
8. 使用者繼續輸入文字、再次 tag 任務、儲存草稿或發布。
9. 使用者從紀錄列表或任務 timeline 回看紀錄時，能理解內容與任務的關聯。

## 5. FMEA UX 風險表

| 失效模式 | 可能原因 | 使用者影響 | 偵測方式 | 優先級 | 通過標準 / 對策 |
|---|---|---|---|---|---|
| 進入任務選取後出現陌生頁面 | selection mode 仍像獨立 picker | 使用者迷路，不知道如何回到紀錄 | 點 `從看板選取` 後觀察畫面 | 高 | 必須停留原看板，只改變選取狀態與提示 |
| 右側欄沒有自動收起 | sidebar state 未同步 | 看板被遮住，難以點任務 | 進入選取模式後看右欄 | 高 | 進入選取模式時右側欄自動收合 |
| 任務 tag 沒插入游標位置 | content cursor offset 遺失 | 內容順序錯，使用者要手動搬移 | 在句中插入任務 tag | 高 | tag 必須插入目前游標位置 |
| tag 看起來像普通文字 | chip 樣式不明顯 | 使用者不知道這是關聯任務 | 視覺檢查內容欄 | 高 | tag 有 chip 樣式、icon、邊界或底色 |
| 點同一任務兩次後摘要重複 | taskLinks 去重失敗 | 關聯清單混亂 | 同任務重複 tag | 中 | 內容可有兩個 tag，摘要只出現一列 |
| 刪除 tag 後摘要仍殘留 | content 與 taskLinks sync 不完整 | AI 關聯錯誤 | 刪除 tag 後觀察摘要 | 高 | 新紀錄中刪除 tag 後摘要同步移除 |
| checklist 子任務不能選 | click handler 只支援 card | 細節任務無法關聯 | 點 checklist item | 中 | card 與 checklist item 都能插入 tag |
| 小視窗中操作按鈕被遮擋 | responsive layout 未覆蓋 | 使用者無法完成流程 | 390x844 或窄窗測試 | 中 | 主要按鈕、內容、tag 不重疊、不溢出 |
| 儲存後重新開啟顯示 raw token | render parser 未套用 | 使用者看到 `@[...](task:id)`，不理解 | 儲存後重新開啟紀錄 | 高 | 一般 UI 顯示 chip 或可讀任務名稱 |
| 選取模式無明確完成/取消路徑 | header/action feedback 不足 | 使用者不知道如何退出 | 進入選取模式後觀察 header | 高 | 有清楚完成或返回控制，且可回到紀錄欄 |

## 6. 實際操作測試案例

| ID | 測試情境 | 操作步驟 | 預期結果 |
|---|---|---|---|
| UX-001 | 開啟會議紀錄 | 在看板點右側 `紀錄`，新增會議紀錄 | 右側欄打開，欄位可填寫，不遮住主導航 |
| UX-002 | 內容輸入 | 在內容輸入 `今天討論 `，游標停在句尾 | 文字正常輸入，placeholder 消失，焦點穩定 |
| UX-003 | 進入看板選取模式 | 點 `從看板選取` | 右側欄自動收起，仍停留原 Kanban 看板 |
| UX-004 | 選取任務卡 | 點一張任務卡 | 不跳頁，任務呈現已選狀態，已選數量更新 |
| UX-005 | 完成選取並插入 tag | 點 `完成` 或完成選取動作回到紀錄欄 | 內容中在 `今天討論 ` 後出現任務 chip |
| UX-006 | tag 後續輸入 | 在 chip 後輸入 ` 的風險` | 文字接在 chip 後，chip 不被破壞 |
| UX-007 | 句中插入第二個 tag | 將游標移到句子中間，再選另一個任務 | 第二個 chip 插入游標位置，不跑到文末 |
| UX-008 | 同任務重複引用 | 同一任務再選一次 | 內容出現第二個相同 chip，摘要仍只有一列 |
| UX-009 | 選 checklist 子任務 | 進入選取模式後點 checklist 子任務 | 子任務也插入 chip，摘要顯示該子任務 |
| UX-010 | 刪除 tag | 在內容中刪除其中一個 chip | chip 可刪除，內容不殘留破碎 token |
| UX-011 | 刪除所有 tag | 刪除內容內所有任務 chip | 新紀錄的關聯任務摘要同步清空 |
| UX-012 | 貼上純文字 | 複製一般文字貼到內容欄 | 只貼入純文字，不帶入奇怪 HTML 或破版 |
| UX-013 | 儲存草稿 | 插入 tag 後點儲存草稿，再重新開啟 | tag 仍可讀，摘要仍保留唯一任務 |
| UX-014 | 發布紀錄 | 插入 tag 後發布，從最近紀錄查看 | 紀錄內容可讀，tag 不顯示 raw token |
| UX-015 | 任務 timeline 回看 | 從被 tag 的任務查看關聯紀錄 | 該任務 timeline 看得到這筆紀錄 |
| UX-016 | 個人工作紀錄時間 | 新增個人工作紀錄 | 記錄人員為自己，時間區間結束為今天、開始為一週前 |
| UX-017 | 會議紀錄參與人員 | 新增會議紀錄並填參與人員 | 參與人員文字可輸入、保存、重開後仍可讀 |
| UX-018 | 取消/返回選取 | 進入選取模式後不選任務，返回紀錄欄 | 原本內容與游標附近文字不遺失 |
| UX-019 | 長任務標題 | 選取長標題任務 | chip 不撐破內容欄，必要時合理截斷或換行 |

## 7. 通過標準

本 DEV 可進入 QC 驗證的 UX 通過標準如下：

- 使用者不需要離開原本 Kanban 看板即可完成任務關聯。
- 進入任務選取時，右側紀錄欄會自動收合。
- 任務 tag 插入位置符合使用者當前內容游標位置。
- 內容可以自然混寫 `文字 + 任務 tag + 文字`。
- 任務 tag 的視覺明確，不像普通純文字。
- 同一任務可在內容中重複被提及，但關聯任務摘要維持唯一。
- checklist 子任務與一般任務卡都能被 tag。
- 儲存或發布後，一般使用者介面不顯示 raw token。
- 在 desktop、laptop、小視窗下沒有主要 UI 遮擋、文字溢出、按鈕無法點擊。

## 8. 證據收集方式

QC 執行時需保留下列證據：

- 每個 viewport 的截圖：
  - 進入選取模式後。
  - 插入 tag 後。
  - 儲存或發布後重新開啟。
- 一段完整流程錄影或連續截圖：
  - 新增紀錄。
  - 輸入文字。
  - 從看板選任務。
  - 插入 tag。
  - 儲存或發布。
- 若有失敗，需記錄：
  - URL。
  - viewport。
  - 操作步驟。
  - 實際畫面。
  - console error。
  - 是否可重現。

## 9. QC 執行建議

建議 QC 先做最小 smoke：

1. 開會議紀錄。
2. 內容輸入一段文字。
3. 從看板選一張任務卡。
4. 完成選取並確認 tag 插入內容。
5. 同任務再選一次，確認內容兩個 tag、摘要一列。
6. 儲存草稿或發布後重新開啟。

建議先執行自動 gate：

```powershell
npm.cmd run verify:dev-003-record-tags
```

此 gate 驗證內容 token helper 的核心行為：游標位置插入、重複 tag 去重、刪除後 structured links 同步、legacy link 保留，以及 preview 不暴露 raw token。

若 smoke 通過，再執行完整 UX-001 到 UX-020。

## 10. 阻塞判定

下列狀況視為 QA/QC 阻塞或重大 UX fail：

- 點 `從看板選取` 後離開原看板，或進入另一個陌生頁面。
- 無法回到原本紀錄填寫狀態。
- tag 沒有插入內容，而只出現在關聯任務摘要。
- tag 固定出現在文末，無法依游標位置插入。
- 儲存後內容消失或 raw token 直接暴露在主要使用者 UI。
- 小視窗下主要操作按鈕不可見或不可點。

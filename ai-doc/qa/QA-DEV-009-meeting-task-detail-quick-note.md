# QA-DEV-009 會議模式任務詳情內快速補記 UX 驗證計畫

狀態：Passed by QC
對應 DEV：DEV-009
建立日期：2026-06-07
驗證主軸：UX / 會議工作流順暢度

## 驗證範圍

- 會議模式下任務詳情快速補記。
- meeting draft append 行為。
- 任務 inline tag 與 `taskLinks` 同步。
- 任務備註與會議補記資料邊界。
- 看板、任務詳情、右側會議速記欄三者的注意力切換成本。
- 桌機與筆電 viewport 下的可見性、可操作性與不重疊。
- DEV-007 / DEV-008 回歸。

## 不驗證範圍

- 手機版會議記錄工作流不作為 DEV-009 通過條件。
- 不要求在 mobile viewport 完成開始會議、補記、發布或會議查找流程。
- 若 mobile smoke 被額外執行，只能作為參考證據，不可作為 release gate。

## UX 驗證目標

- 使用者在會議中討論某任務時，不需要離開任務詳情即可完成補記。
- 使用者能明確分辨「本次會議補記」與「任務備註」不同資料用途。
- 補記動作要短、直接、可預期：輸入、送出、清空、可在會議紀錄看到。
- 看板仍是主畫面；任務詳情不能讓使用者迷失在紀錄頁或完整編輯器中。
- 桌機與筆電下不可出現文字裁切、按鈕不可按、輸入框被遮住或水平 overflow。

## UX 假設

- 主要使用者是會議中的記錄者或專案負責人。
- 會議中使用者會同時看任務狀態、日期、指派人與補記討論內容。
- 「任務備註」是長期任務資料；「本次會議」是當次 meeting draft 的一段紀錄。
- 第一版不要求在任務詳情內編輯整篇會議紀錄，只要求快速補記。

## 使用者關鍵流程

- 開始會議後，在看板打開任務詳情。
- 在任務詳情內輸入本任務討論內容並加入紀錄。
- 發布或儲存前，meeting draft 已包含該任務討論。
- 未來從任務知識可查到該片段。
- 中途修改任務狀態、日期或指派人，不影響補記輸入。
- 補記後可立即繼續看下一張任務卡，不需要先去紀錄庫或完整紀錄頁。

## FMEA 風險表

| 失效模式 | 原因 | 影響 | 偵測方式 | 優先級 | 對策 |
|---|---|---|---|---|---|
| 補記寫進任務備註 | 混用 `detailNotes` 與 record draft | 會議來源與時間脈絡遺失 | 驗證 append 後 `detailNotes` 不變 | 高 | 只透過 record store action 寫 meeting draft |
| 沒有 task tag | 只 append 純文字 | DEV-008 無法依任務查找 | 檢查 draft content token | 高 | action 自動插入 `serializeTaskMention` |
| 非會議模式誤顯示 | UI 條件不足 | 一般任務編輯被干擾 | 非 meeting mode UI smoke | 中 | 只在 `isMeetingMode` 顯示 |
| 空白也可加入 | 未 trim | 會議紀錄污染 | 空白輸入測試 | 中 | 空白內容 no-op |
| 重複 header 或 section 混亂 | append 規則不穩 | 紀錄可讀性下降 | 多次 append 測試 | 中 | 寫入同一 `## 任務討論` 區塊 |
| 使用者分不清「本次會議」與「備註欄」 | 標題、位置或視覺層級不清 | 把會議內容寫錯地方 | 觀察使用者是否能第一次正確選入口 | 高 | 「本次會議」放在備註欄前，使用藍色會議區塊 |
| 補記後不知道是否成功 | 無清楚回饋或內容不出現在 draft | 使用者重複輸入或不信任系統 | 送出後檢查輸入框清空與 meeting draft 出現內容 | 高 | 清空輸入框，右側 draft 可看到補記 |
| 補記區搶走任務編輯焦點 | 區塊過大或位置干擾狀態/日期 | 會議中改任務變慢 | 桌機流程觀察狀態、日期、指派、補記連續操作 | 中 | 保持 compact 區塊，不取代任務基本欄位 |
| 筆電 viewport 補記區被裁切 | modal 高度與右側速記欄同時存在 | 使用者補記時需要額外關閉/重開面板 | 1024x768 viewport 實測 | 高 | modal 需可捲動，補記區與按鈕可見或可捲動到 |
| 長文字造成版面破裂 | 會議補記可能含長 URL / 英文 / 數字 | modal overflow 或按鈕被擠壓 | 輸入長字串與多行文字 | 中 | textarea 可換行，容器不可水平 overflow |

## 測試案例

- TC-001：非 meeting mode 開任務詳情，不顯示「本次會議」。
- TC-002：meeting mode 開任務詳情，顯示快速補記 textarea 與「加入紀錄」。
- TC-003：輸入文字並送出，meeting draft content 包含 `## 任務討論` 與 `@[title](task:id)`。
- TC-004：送出後輸入框清空，任務備註內容不變。
- TC-005：空白內容送出不改 draft。
- TC-006：連續兩次補記會寫入同一任務討論區塊。
- TC-007：`Ctrl+Enter` 可送出。
- TC-008：DEV-007 meeting activity append 仍可運作。
- TC-009：DEV-008 任務知識片段抽取仍可讀到補記內容。
- TC-010：UX 主流程，使用者從看板開任務詳情、改狀態、補記、關閉詳情，過程不可跳到紀錄頁。
- TC-011：補記成功回饋，送出後 textarea 清空，右側 meeting draft 或紀錄內容可找到補記文字。
- TC-012：資料邊界，補記送出後任務備註 textarea 不應被自動填入該會議文字。
- TC-013：長文字，輸入 120 字中文、長英文單字、URL 與多行內容，modal 不得水平 overflow。
- TC-014：筆電 1024x768，打開任務詳情後可看到或捲動到「本次會議」、textarea 與「加入紀錄」。
- TC-015：視覺層級，使用者應能一眼分辨「本次會議」是當次紀錄入口，「備註欄」是任務長期資料。
- TC-016：錯誤防呆，空白內容時「加入紀錄」不可用或送出 no-op。
- TC-017：連續任務，補完任務 A 後關閉詳情，打開任務 B 可補記 B，兩者 task tag 不混淆。

## UX 驗證腳本

### Script A：桌機主流程

1. 開啟 `http://127.0.0.1:4173/`。
2. 使用固定測試環境登入。
3. 點「會議紀錄」進入 meeting mode。
4. 在看板右鍵任務「品質驗證測試任務 1」，點「更多詳情選項」。
5. 確認任務詳情內可見「時間設定」、「本次會議」、「備註欄」、「任務知識」。
6. 修改狀態或日期任一欄位，再在「本次會議」輸入 `UX smoke 任務補記`。
7. 點「加入紀錄」。
8. 確認 textarea 清空，右側 meeting draft 可找到 `UX smoke 任務補記`，並包含目前任務 tag。
9. 關閉任務詳情，確認仍停留看板。

### Script B：資料邊界

1. 在 meeting mode 任務詳情輸入一段補記。
2. 送出後檢查「備註欄」內容未被自動改成該補記。
3. 檢查 meeting draft content 包含 `## 任務討論`。
4. 檢查 task links 包含目前任務。

### Script C：筆電 viewport

1. 設定 viewport `1024x768`。
2. 進入 meeting mode 並開啟任務詳情。
3. 確認「本次會議」標題、textarea、「加入紀錄」可見且可操作。
4. 輸入多行文字並送出。
5. 確認沒有水平捲動、遮蔽、裁切或按鈕不可按。

## Viewport 與互動狀態矩陣

| Viewport | 必測狀態 | 判定 |
|---|---|---|
| 1440x950 desktop | meeting mode、任務詳情開啟、右側速記欄開啟 | 不遮蔽主要任務欄位；補記可送出 |
| 1024x768 laptop | meeting mode、任務詳情開啟 | modal 可捲動，按鈕不裁切 |

## 通過標準

- `verify:dev-009-task-detail-quick-note` 通過。
- lint、DEV-007、DEV-008、build 通過。
- Playwright smoke 確認 meeting mode 任務詳情有快速補記 UI。
- 桌機與筆電主流程均未離開看板或跳到紀錄頁。
- 使用者可清楚分辨「本次會議」與「備註欄」。
- 補記成功有可觀察回饋：輸入框清空，meeting draft 可見新增內容。
- 無重疊、裁切、不可讀、不可操作、非預期水平 overflow。

## 失敗時需收集證據

- 截圖：失敗前、失敗當下、失敗後。
- Viewport 尺寸與 URL。
- 操作步驟與實際點擊/輸入內容。
- Console error / warning。
- meeting draft content 摘要。
- 任務備註欄內容截圖或 DOM snapshot。
- 若是視覺問題，標記重疊、裁切或 overflow 的元件位置。

## 證據收集方式

- 指令輸出：DEV-009 verifier、lint、回歸 verifier、build。
- UI smoke：固定測試環境中進入會議模式後，任務詳情顯示「本次會議」快速補記。
- Playwright snapshot：桌機與筆電各至少一份。
- 截圖：桌機成功補記後、筆電 viewport 顯示補記區。

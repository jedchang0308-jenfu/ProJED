# ProJED 開發邏輯規範 (Reference: Trello)

此 Skill 定義了 ProJED 專案的開發原則。除非使用者另有指定，否則所有的功能開發、UI 設計及互動邏輯都必須參考 **Trello** 的標準建立。

## 1. 核心階層結構
1.  **看板 (Board)**：最高層級，包含多個列表。
2.  **列表 (List)**：橫向排列的內容容器，可包含多張卡片。
3.  **卡片 (Card)**：具體的工作單元，包含標題、日期、標籤、附件、備註及待辦清單。
4.  **待辦清單 (Checklist)**：卡片內的細分項目。

## 2. 邏輯規範 (Logic Rules)
- **拖拽排序 (Drag & Drop)**：所有的列表、卡片、待辦項目都應該支援直覺的拖拽排序。
- **詳情管理 (Modal UI)**：點擊卡片後應彈出詳情視窗 (Modal)，在視窗內進行大部分的編輯操作。
- **即時保存 (Auto-save)**：所有的修改（標題、日期、內容）應在失去焦點或操作完成時立即保存並同步。
- **狀態聯動**：
    - 當卡片的所有待辦項目完成時，應考慮建議使用者更新卡片狀態。
    - 列表的名稱應可直接點擊修改。

## 3. 視覺與體驗 (UX/UI)
- **Premium 質感**：使用現代字體 (Inter, Outfit)、柔和的深色模式 (Dark Mode)、細微的過渡動畫和陰影。
- **互動反饋**：懸停特效、點擊反應、Toast 通知。
- **簡潔直覺**：介面應保持專注，避免過多的雜訊。

## 4. 技術規範
- **核心架構**：Vanilla JS (app.js) + CSS (style.css)。
- **持久化**：
    - `localStorage` (本地緩存)
    - `Firebase Firestore` (雲端同步)
    - `Google Calendar` (日期同步)
- **程式碼註解**：每一段複雜的邏輯都必須使用 **繁體中文** 進行詳細註解。

## 5. 開發工作流
- 修改程式碼後，必須考量對看板視圖與甘特圖視圖的同步影響。
- 優先確保資料結構的一致性，防止 ID 衝突或資料遺失。

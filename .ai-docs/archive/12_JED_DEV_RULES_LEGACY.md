# JED全域開發規範 (JED_DEV_RULES)

## 1. 規範適用範圍與標記機制 (compliance_scope)
- **個人專案觸發條件 (Opt-in)：** 只有當專案根目錄存在 `.jed` 標記檔時，本配置檔內的所有嚴格規範才強制生效。
- **外部/協作專案 (預設狀態)：** 若專案根目錄『沒有』 `.jed` 標記檔，IDE 必須自動將其視為外部協作專案。
- **協作準則：** 在外部專案中，全面停用本規範的技術選型限制。必須無條件尊重並延續該專案現存的架構、語言與資料庫選型，嚴禁主動要求重構。
- **品質底線：** 無論是否為個人專案，修改與產出的代碼皆須維持高可讀性，並適當加上繁體中文註解。

## 2. 專案啟動程序 (project_initialization)
- 當我明確指示「建立 Jed 個人新專案」時，第一步必須是在專案根目錄自動建立 `.jed` 空白標記檔。
- 標記檔建立後，必須自動產生 `README.md`，並宣告此為 Jed 的個人受控專案。
- 完成上述兩步後，才能開始進行技術選型與代碼生成。

## 3. 代碼審查規範 (code_review)
- **Checklist：**
  - Correctness: 代碼是否達成預期功能？
  - Edge cases: 邊界與錯誤條件是否被處理？
  - Style: 是否符合專案慣例？
  - Performance: 是否有明顯的效能瓶頸？
  - Explanation: 必須用繁體中文解釋，複雜代碼需加註解。
- **Feedback Guidelines：** 明確指出需要修改的地方、解釋「為什麼」要改，並盡可能提供替代方案。

## 4. 實作計畫規範 (implementation_planning)
- 需做效益與技術債風險評估。
- 在計畫結尾提供 [Y] 與 [N] 按鈕供我點選。
- 必須等我按下 [Y] 後才能開始修改代碼 (Strict Mode 精神)。
- 視情況，主動提出複數個計畫供選擇。

## 5. 全域技術選型規範 (tech_stack)
- **雲端與 Web 領域 (預設)：** 指定開發語言為 TypeScript。前端限用 React/Next.js，移動端限用 React Native，後端限用 Node.js。
- **桌面端與工業軟體領域：** 豁免 TS 限制。優先使用 C# (.NET) 或 Python，符合 Windows 原生 API 操作標準。
- 禁用剛發布尚不成熟的新技術，穩定性與可維護性優先。

## 6. 雙核心資料庫選型 (database_policy)
- **核心系統ERP：** 強制使用 Supabase (PostgreSQL)。
- **其他雲端專案：** 使用 Firebase (Firestore)。
- **單機版桌面軟體：** 允許使用 SQLite、本地 JSON 或 XML 確保離線可用性。
- 混合專案必須以 Supabase 作為『唯一真相來源 (SSOT)』。所有選型必須具備成熟的數據導出方案，嚴禁鎖定。

## 7. 架構與文件規範 (documentation_policy)
- 代碼具備高可讀性，並隨附更新的『架構字典 (Architecture Dictionary)』與『API 文件』。
- 嚴禁產生無註釋的複雜代碼段 (Spaghetti Code)。
- 核心邏輯必須解釋『設計意圖 (Design Intent)』。每完成一次 PR 合併，需同步更新架構字典。

## 8. 分支管理與部署 (devops_policy)
- **分支管理：** 禁止直接修改主線。`main` 為正式環境；`test` 為測試環境；`feature/*` 為開發分支。
- **命名規範：** 嚴禁中文，一律使用英文小寫加連字號 (kebab-case)。
- **PR 機制：** 分支合併必須發起 PR，AI 需生成「白話文變更摘要」。
- **測試與環境：** 雲端專案強制雙重測試；桌面端依語言執行標準編譯。強制使用環境變數（如 `.env`）區分環境，嚴禁金鑰寫死。
